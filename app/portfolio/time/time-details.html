Employee Vacation Tracker

The Employee Vacation Tracker allows employees to request time off through a simple interface, as well as view the time off of their team.

I was the lead designer on this project. In addition to completing a heuristic review of the existing request system, I interviewed many users of the current site to understand their pain points and difficulties in preparation for designing the new tracker.

Due to my non-disclosure agreement, I will not be sharing actual images of the site as it was. Rather, all opinions and words are my own, and all prototypes are my own based on the original.

1| Heuristic Evaluation of Existing Site

Initial Setup

The initial setup information seems like something we should already know about the user and allow them to confirm or update during setup. Furthermore, many employees may not know how many carryover days they have immediately.

For a global corporation, the list of work locations to select from is sparse and confusing. Certain items, such as 'State of Massachusetts' is not uniform with the city, state format, and 'San Francisco, CA' and 'California - not including San Francisco' is confusing as well.

Home

I could change how the user information is displayed so there are more logical groupings of information.

  • Name, Employee ID, Location
  • Employee Status, FSLA Status, Standard Hours, Standard Hours Effective Date
  • Time Record End Date, Time Record Begin Date, Pay Cut Off Time
  • Unclear is this information is necessary for an average full-time employee

It is also confusing to have a message stating that you cannot use the system as an exempt employee.

My Attendance

It would be nice to see employee request information in a more visual representation. The status of my pending request and teh new/edit request screen should be more noticeable than my employee information on this screen.

Save Request

I expect to see "Submit" as an option within the confirmation modal, rather than 'Confirm' and 'Reject' with the same visual treatment.

Error Screen

The error message is not entirely obvious on the page.

Success/Confirmation Screen

The vacation code is not commonly understood by everyone. This should be listed in full text.

There is not a lot of visual feedback on the screen to indicate that the request status changed

Cancelled requests should be removed from the approvals list.

Should last modified time/by be combined? Unless the user is able to sort, this may not be necessary. How are these columns used? If I am unable to edit requests, these columns will never update.

New Request

It would be nice to see a balance of vacation days as I am choosing the vacation code.

Does it make sense for the default code to be Bereavement? It would make more sense to default to "Vacation" or simply "Choose Time Off Type".

Vacation Balance

This page is not easily scanned by the user. The two most important numbers the user cares about are how much vacation they have for the year, and how much they have left to use.

My History

This page only shows how many days are available, but it would be nice to give the user some information about when they used those dates.

Time

This entire screen is meant for non-exempt employees, however there is no indication of this on the screen.

2| Empathize with the User

User Interviews

I gathered the following candid feedback by utilizing our internal social networking site. Below is a compiled list of notes, quotes, and general findings from speaking with users of the system:

  • "I tried to request 8 days off, from a Wednesday to Friday of the following week. This should have been 8 days off. The form reported 10 days as the total number of days since it counted the weekend. This stopped me from submitting the request until I find out if that is how the calculation is done. It would be a poor design if I have to submit these as 2 separate requests if the system does not know about weekends.
  • "There is no option for volunteer days."
  • "There is no way a few hours off, only a half day."

  • "Managers should be able to input absences on behalf of their employees."
  • "When an employee enters a request, "To Date" should prepopulate with the same date as the "From Date". They shouldn't have to select "To Date" for a single day absence."
  • "There are so many different Vacation Codes, why are Service Recognition and Voluntary Time off in the same option? I don't see what they have to do with each other. Service Recognition is earned. I don't know what Voluntary Time off means. Isn't ALL time off voluntary?"

  • "What are some things you think work well about the current submission process?"
  • "No way to sort or filter absence requests."

User Research

After choosing the card model, I conducted users quick usability tests with users that had previously provided feedback. The following questions were asked:

  1. Do you have direct reports? If so, how many?
  2. How large is your team/group of people you regularly work with in an average week?
  3. Are there any factors related to your team that would affect how you plan your vacation?
  4. What information about you or your team's upcoming vacation might you be interested in seeing on your homepage?
User Question 1 Question 2 Question 3 Question 4
User 1 User does not have direct reports. User works with a team of 40-50 people each week, including offshore employees. Schedule is not dependent on other team members, so vacation is taken whenever needed. User does plan vacation ahead of time. User is less interested in viewing upcoming team members' vacation, however finds it useful in scheduling meetings. In planning own vacation, user is interested in clearly seeing how many days are available. Same for manager's perspective in approving vacation requests without having to visit a separate screen. Managers should be able to see whole team's schedule in one place. Clean, simple interface.
User 2 User has 7 direct reports and is responsible for the approval of 8 more direct reports for immediate manager. Aside from the 15 employees (all in same location) requiring user's approval, user also works with many different teams on different schedules totaling upwards of 100 people each week on average. Previously - In the beginning of the year, a paper is passed around for employees to request initial vacation, based on seniority. Currently - Process is still based on seniority, however user must sit down with employees to ensure this still stands User is interested in seeing a simple summary of own vacation (how many days taken/requested/available). For team, user is interested in seeing a simple summary of team absences for the current week with labels for the type of absence.
User 3 User has 18 direct reports. In an average week, user typically works with 4 of 18 direct reports regularly. Due to the nature of the team, vacation is usually consistent from year to year, aside from various personal days.
Currently, user has a physical color-coded calendar to plan/show when out of office. 		User is interested in seeing a team view of absences more so than information about own vacation. User is interested in determining overlapping coverage, and would like to receive an alert if there is a conflict. User believes team should be able to see same information as manager. User would like to see different colors for different absence reasons. User would like notification of local Sick Day Policy when approving or viewing team vacation, especially for remote managers (not necessary for entire team to view). User would not find it useful to view team's historical absences.
User 4 User does not have direct reports. User constantly works with different teams. These are typically small groups, but sometimes can be very large. Vacation is not dependent on other team members, but generally the team tries to make sure there is coverage at all times. For self, user is interested in viewing a simple calendar view. User often "Works out Loud". User wants to see holidays and be able to move through the calendar.
User 5 User has 6 direct reports, not including 2 offshore. Two are in the same office, two are in a different office, two work remotely. On a given week, user works with about 15 people from various teams. Team has very little overlap when requesting time off. Previously, team used a spreadsheet to track time off. Team prefers MyTime over PPM. Team keeps physical calendars to mark Out of Office. User is not interested in seeing personal time off at a glance. User would like to see team absences about a week out, with the reason displayed over the absence bar. User wants to see manager and peers' schedules, with an easy method of viewing absences within different working groups. User does not see value in selecting holiday schedules in calendar because those tend to be more permanent.

Defining User Personas

Through user interviews, we identified two primary personas:

1| Submitter

Primary Tasks

  • Request time off through the portal
  • Ability to view upcoming time off and days remaining

Secondary Tasks

  • Ability to view team's upcoming time off

2| Approver

Primary Tasks

  • Ability to review and approve time off requests
  • Ability to view direct reports' upcoming time off

Secondary Tasks

  • All tasks of Submitter persona

3| Define the Problem

After defining the user personas, the problem statements can be defined as follows:

  1. Submitters need to be able to easily view upcoming time off, as well as request time off.

  2. Approvers need to be able to easily view direct reports' vacation requests in order to approve vacation requests with minimal overlap.

4| Ideate Possible Solutions

After listening to feedback from users, I drew out screen ideas for what the redesigned application could look like.

Features

After iterating through possible views, I ultimately decided it would be best to go with a simple dashboard widget that users can add to their Intranet Home Page. The majority of the functionality a user needs does not need to be contained in a full site that may assume the user spends more time requesting vacation than they do. Requests can happen from a modal window, same for approvals. This way, the application is as simple as possible.
